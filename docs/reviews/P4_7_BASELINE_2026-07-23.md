# P4.7-0 Baseline Observation 2026-07-23

> Document status: partial P4.7-0 observation, not a performance gate.
> The formal baseline still requires full-column traversal, cache counters, allocation measurement and four-combination JFR.

## Fixture

- Profile: non-persisted P4.6 archipelago smoke profile.
- Seed: `123456789`.
- Window: 16 x 16 columns beginning at `(8192, 8192)`.
- Runtime path: `OUTER_CONTINENTS`, `WITH_FLOOR`, `REGION_PLANNED`, archipelago enabled.
- Operation: `EndHeightmap.sampleTerrainProfile` into one caller-owned `EndTerrainProfileBuffer`.
- Warmup: 1 chunk ordered + 1 chunk deterministic shuffled.
- Measurement: 4 chunks per traversal, 1024 profile samples.

## Observation

One local JDK 21 run produced:

| Traversal | Time |
| --- | ---: |
| Ordered columns | 60,385.9 ns/profile |
| Deterministic shuffled columns | 32,033.3 ns/profile |

Both traversals produced the same checksum: `2755942566284`.

The checksum equality is now a test assertion. The timing difference is evidence for further cache, branch and JIT investigation; it is not a claim that shuffled access is intrinsically faster.

## Cold/Warm Column Latency

The same fixture was measured with 24 full-column chunk traversals. A cold sample
creates a new `EndDensity` owner before the traversal; a warm sample reuses one
owner after an initial traversal. One full `:common:test` run on JDK 21 produced:

| State | p50 | p95 |
| --- | ---: | ---: |
| Cold owner | 2.115 ms | 3.872 ms |
| Warm owner | 1.452 ms | 3.161 ms |

The test asserts that cold and warm traversals have identical density checksums.
These are local observation values, not hardware-independent thresholds and not
evidence about C2ME scheduling, allocation rate, client rendering or JFR stalls.

## Local Analytical Candidate Cost

The P4.7a local analytical candidate was run against the canonical ten-fixture
set (33 x 33 samples, two-cell halo, 4-block stencil, caller-owned output). One
full `:common:test` run on JDK 21 produced `39.1 ns/sample` with checksum
`2369538144867456`.

This measures only the candidate's local primitive computation. It excludes raw
profile sampling, column-cache refresh, allocation accounting, NoiseChunk,
C2ME scheduling and client rendering; it is an observation, not a gate.

## Column Cache Observation

The same 16 x 16 window was sampled through `EndDensity` at five Y values per column. The test-only per-thread cache counters reported:

| Metric | Ordered | Deterministic shuffled |
| --- | ---: | ---: |
| Requests | 1,280 | 1,280 |
| Hits | 1,024 | not separately recorded |
| Misses | 256 | 256 |
| Collisions / evictions | 82 / 82 | 82 / 82 |
| Owner swaps | 1 | not separately recorded |
| Full-column refreshes | 256 | not separately recorded |

Both access orders produced identical density checksums. The observed `82` collisions confirm that a 256-entry direct-mapped cache does not give one independent slot to every column in a 16 x 16 traversal. The counters are disabled by default, use no atomics, and exist only as a test/dev diagnostics seam.

## Limits

This observation still does not measure allocated bytes, raw-top evaluation count
below the full-column boundary, tile construction, client rendering or
ETF/RTF/C2ME JFR combinations. P4.7-0 remains open until those measurements and
the four-combination client/server evidence exist.
